# Bijux GitHub Governance Pack (Python)

Canonical source-of-truth payload for Python-first Bijux repositories.

Current standards orchestration for consumers is driven from
`bijux-std/.github`:

- typed manifest:
  `.github/standards/repo-config.manifest.json`
- workflow inventory registry:
  `.github/standards/workflow-inventory.json`
- renderer:
  `.github/scripts/render_repo_configs.py`
- sync orchestrator:
  `.github/scripts/sync_github_standards.py`

This `shared/bijux-gh` subtree remains as legacy source content for
cross-repository governance assets that are also promoted into the canonical
`.github` standards layer.

Live GitHub admin settings such as branch protection are governed in
`bijux-iac`, not from this subtree.

- `.github/automation-identity.md`
- `.github/CODEOWNERS`
- `.github/ISSUE_TEMPLATE/*`
- `.github/PULL_REQUEST_TEMPLATE/*`
- `.github/scripts/wait_for_ci.py`
- `.github/workflows/deploy-docs.yml`
- `.github/workflows/release-artifacts.yml`
- `.github/workflows/release-crates.yml`
- `.github/workflows/release-ghcr.yml`
- `.github/workflows/release-github.yml`
- `.github/workflows/release-pypi.yml`
- `.github/required-status-checks.md`
- `.github/rulesets/main-branch-protection.json`
- `.github/dependabot.yml` generated by
  `.github/scripts/render_repo_configs.py` from typed manifest inputs

Usage in a repository (current model):

1. Keep standards synchronized from `bijux-std` via
   `.github/scripts/sync_github_standards.py`.
2. Render repository-local wrappers and config from the typed manifest via
   `.github/scripts/render_repo_configs.py`.
3. Keep managed runtime workflow copies filtered by repository
   `workflow_allowlist` from the manifest.
4. Enforce drift/policy checks via:
   - `.github/workflows/bijux-std.yml`
   - `.github/workflows/github-policy.yml`

This keeps in-repo GitHub standards and bot update policy consistent across
repositories. Live branch protection is enforced separately by `bijux-iac`.

## Shared Scripts

Canonical reusable release helper scripts are provided under `.github/scripts`.

- `wait_for_ci.py`: waits for the configured CI workflow to complete for the release SHA.

Repository-specific release prepare scripts may still exist in `.github/scripts`, but shared helpers should come from this canonical source.

## `release-pypi.yml` configuration contract

The shared PyPI workflow supports two publication modes behind one contract:

- `maturin`: build and publish a Rust-backed Python bridge package.
- `artifact`: publish previously staged wheel/sdist artifacts (typically from a
  repository-level `publish.yml` build job).

Configuration sources (highest precedence first):

1. `workflow_call` / `workflow_dispatch` inputs
2. `.github/release.env`
3. repository variables (`Settings -> Secrets and variables -> Actions -> Variables`)
4. built-in defaults

Common keys:

- `BIJUX_PYPI_ENABLED` (`true|false`)
- `BIJUX_PYPI_MODE` (`maturin|artifact`)
- `BIJUX_PYPI_PACKAGE_MATRIX_JSON` (JSON array for artifact mode)
- `BIJUX_PYPI_ALLOWED_PACKAGES` (space/comma/semicolon-separated package slugs)
- `BIJUX_PYPI_ENVIRONMENT_NAME` (default `pypi`)
- `BIJUX_PYPI_PUBLISH_AUTH_DEFAULT` (default `trusted`)
- `BIJUX_PYPI_SKIP_EXISTING` (default `true`)

Maturin mode keys:

- `BIJUX_PYPI_WAIT_FOR_CI_COMMAND`
- `BIJUX_PYPI_PLAN_COMMAND`
- `BIJUX_PYPI_PREPARE_COMMAND`
- `BIJUX_PYPI_PUBLISH_COMMAND`
- `BIJUX_PYPI_SETUP_PYTHON`, `BIJUX_PYPI_PYTHON_VERSION`
- `BIJUX_PYPI_SETUP_RUST`, `BIJUX_PYPI_RUST_TOOLCHAIN`
- `BIJUX_PYPI_MATURIN_VERSION`
- `BIJUX_PYPI_CACHE_DEPENDENCY_PATH`
- `BIJUX_PYPI_MANIFEST_PATH`
- `BIJUX_PYPI_INTERPRETER`

## Workflow Naming Convention

Use `ci.yml` as the primary CI workflow filename in each repository.

- Reusable CI workflows should also use `ci.yml` when they are the main CI contract.
- Keep repository-specific workflow names descriptive (for example `ci-canon.yml`) and avoid cross-repository product names in unrelated repositories.

## `deploy-docs.yml` configuration contract

The shared workflow is placeholder-driven. Consumers can override behavior through:

- repository variables (`Settings -> Secrets and variables -> Actions -> Variables`)
- optional `.github/docs-deploy.env` in the repository
- example template: `workflows/deploy-docs.env.example`
- adoption checklist for Bijux repositories: `workflows/deploy-docs-adoption.md`

Environment values in `.github/docs-deploy.env` override repository variables.

Supported keys:

- `BIJUX_DOCS_SITE_URL` (default: inferred from repository name)
- `BIJUX_DOCS_SITE_DIR` (default: `artifacts/docs/site`)
- `BIJUX_DOCS_INSTALL_COMMAND` (default: auto-discover make target)
- `BIJUX_DOCS_BUILD_COMMAND` (required if auto-discovery cannot find a build target)
- `BIJUX_DOCS_VERIFY_COMMAND` (optional)
- `BIJUX_DOCS_SETUP_PYTHON` (`true|false`, default: auto-detect)
- `BIJUX_DOCS_SETUP_UV` (`true|false`, default: auto-detect)
- `BIJUX_DOCS_SETUP_NODE` (`true|false`, default: auto-detect)
- `BIJUX_DOCS_SETUP_RUST` (`true|false`, default: auto-detect)
- `BIJUX_DOCS_PYTHON_VERSION` (default: `3.11`)
- `BIJUX_DOCS_NODE_VERSION` (default: `22`)
- `BIJUX_DOCS_RUST_TOOLCHAIN` (default: `stable`)

The workflow validates and uploads the first detected directory containing
`index.html` from this priority list:

1. `BIJUX_DOCS_SITE_DIR` (if set)
2. `artifacts/root/docs/build-site`
3. `artifacts/site/bijux-masterclass`
4. `artifacts/docs/site`
5. `artifacts/root/docs/site`
