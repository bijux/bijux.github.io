Platform model

One engineering center, multiple product lines

bijux-core is the engineering center of gravity. Around it sit product lines for AI workflows, data delivery, proteomics, genomics, pollenomics, learning, and GNSS. The shape is deliberate: one runtime foundation, several focused applications.

Consolidated center

The runtime, DAG engine, evidence model, release contracts, and maintainer tooling now live in one canonical repository.

consolidation: bijux-cli + bijux-dag now presented as bijux-core

Clear product boundaries

The surrounding systems are distinct products with their own technical roles, linked by shared engineering discipline instead of blurred into one catch-all codebase.

consolidation: bijux-rag, bijux-rar, bijux-vex, bijux-agent, and agentic-flows now read as the canon product family

Best for: teams that need a runtime backbone for governed execution, evidence, release control, and pipeline lifecycle management.

bijux-core combines the operator-facing CLI, DAG semantics, replay, evidence, and maintainer workflows into one source of truth for execution-heavy systems.

  • Operator runtime through bijux-cli
  • DAG definition, execution, replay, and diff through the unified DAG surface
  • Maintainer control-plane tooling for diagnostics, governance, contracts, and release checks

bijux-dag is the upcoming public DAG surface targeted for v0.4.0: a model for pipeline identity, replay, diff, lineage, and semantic change rather than task scheduling alone.

Why it matters: it gives the rest of the platform a serious execution backbone and a clear path for reproducible pipeline change control.

Principles

Engineering principles

The work is designed around repeatability, inspectability, and explicit boundaries across runtime systems, AI workflows, scientific software, and release surfaces.

Contract-first architecture

APIs, schemas, and operational surfaces are treated as versioned contracts, not incidental implementation details.

Deterministic execution

Runtime systems prioritize replayable behavior, controlled side effects, and evidence-rich outputs suitable for governance workflows.

Cross-language depth

Rust and Python are used intentionally across product surfaces, with clear package boundaries and reproducible release paths.

Platform coherence

Repository topology, docs, automation, and release metadata are aligned so platform evolution stays legible at scale.
